<tt><font size="2">> <br>
> There are a few things in the current ABI which are known to be <br>
> suboptimal, but we cannot change because doing so would introduce an<br>
> ABI break. However, vendors sometimes get an opportunity to break <br>
> their ABI (or are defining a new ABI), and for some vendors, this is<br>
> a very common occurrence. To this end, I think it would be valuable <br>
> for the ABI document to describe what we might want to put in a <br>
> 'Version 2' of the ABI; that is, a set of changes that we recommend <br>
> be made whenever a vendor has a chance to introduce an ABI break.</font></tt><br>
<tt><font size="2">> <br>
> (Or perhaps this should be viewed from the opposite perspective: we <br>
> could make improvements to the ABI, with an annex listing changes <br>
> that old platforms must make for compatibility.)</font></tt><br>
<tt><font size="2">> <br>
> Would there be support for this idea?</font></tt><br>
<tt><font size="2">> <br>
> In off-line discussion with John McCall, we came up with the <br>
> following list of potential changes that might be made (sorry if I <br>
> forgot any):</font></tt><br>
<tt><font size="2">> <br>
>  * Make constructors and destructors return 'this' instead of <br>
> returning 'void', in order to allow callers to avoid a reload in <br>
> common cases and to allow more tail calls.</font></tt><br>
<br>
<tt><font size="2">This seems like a simple idea, but does it save extra calls in every case for the cost of the ABI breakage? I think I can think of a few cases where it will save, but can there be cases where it won't save? Do we think we will implement this using another set of differently named C1,C2, C3, D1,D2,D3 to make sure that we know which one return this and which one return void, otherwise how do implementations tell the difference between the old set and the new set?</font></tt><br>
<br>
<tt><font size="2">>  * Simplify case 2b in non-POD class layout.</font></tt><br>
<tt><font size="2">I could not track down what the simplify proposal is in the ARM C++ ABI. Is this saying we will remove the complication of the search for the Nearly-Empty base which causes such headache for implementers 10 years ago? But now that we all have it, is it worth the few weeks that it would take to pull it out. For us, this code is fairly infused into the whole primary base search mechanism. Is this simplification something else?</font></tt><br>
<br>
<tt><font size="2">>  * Make virtual functions that are defined as 'inline' not be key functions</font></tt><br>
<br>
<tt><font size="2">While I like this change because it removes a persistent ambiguity, my question is do the majority of compilers pick the base that does not have the out-of-class inline keyword (i.e. the first virtual void f()). We still pick the out-of-class inline one (the virtual void b()) that is not declared inlined in the class. This is clearly IMHO against the spirit of the key function, but don't all compilers do the same thing and if so, why change it? May be it is better to change the definition to define precisely what everyone already do and just leave it as a wart?</font></tt><br>
<br>
<tt><font size="2">>  * Fix the bug that -1 is both the null pointer-to-data-member value<br>
> and also a valid value of a pointer-to-data-member (could use <br>
> SIZE_MIN instead)</font></tt><br>
<br>
<tt><font size="2">This is the one I would probably pick to like the most if someone can show me the ambiguous case. I just could not think of it. Aren't all the offset to base always positive? Since you said it exists, I believe you so I just want to see what it is and if so I would agree the fix is needed.</font></tt><br>
<br>
<tt><font size="2">>  * Relax the definition of POD used in the ABI, in order to allow <br>
> more class types to be passed in registers</font></tt><br>
<tt><font size="2">> <br>
> Are there any other things that it would make sense to change in a <br>
> version 2 of the ABI?</font></tt><br>
<br>
<tt><font size="2">Do we want to specify how dynamic Thread local storage is done? As some can do it using a Compiler guard, while others can do it through a new OS section making it non-interopearable?</font></tt><br>
<br>
<tt><font size="2">> <br>
> Also, would there be any support for documenting common deviations <br>
> from the ABI that platform vendors might want to consider when <br>
> specifying their own ABIs? In addition to some of the above, this <br>
> would also include:</font></tt><br>
<tt><font size="2">> <br>
>  * Representation of pointers-to-member-functions (in particular, <br>
> the current representation assumes that the lowest bit of a function<br>
> pointer is unused, which isn't true in general)</font></tt><br>
<tt><font size="2">>  * Representation of guard variables (some platforms use the native <br>
> word size rather than forcing this to be 64 bits wide)</font></tt><br>
<tt><font size="2">> <br>
