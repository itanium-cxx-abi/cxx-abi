This change doesn't seem to have been applied to the ABI, though G++ implements it.  What are other compilers doing?</blockquote><div><br></div><div>I happened to be looking into this very recently... Clang implements the ABI as currently specified (along with the corresponding miscompiles). However, the proposed approach doesn't appear to remove all the miscompiles, for at least two reasons:</div>


<div><br></div><div>1) A parameter can be passed or a return value constructed by calling a constructor which is not a copy or move constructor. This can happen when a templated constructor is selected:</div><div><br></div>

<div>
struct A {</div><div>  A(const A&) = default; // suppress implicit move constructor, still trivial</div><div>  template<typename T> A(T &&); // chosen for copies from non-const objects and for moves, not a copy constructor nor a move constructor</div>

<div>
};</div><div><br></div><div>2) A parameter can be passed or a return value constructed by copy-list-initialization with no copy or move occurring:</div>
<div><br></div><div>struct B {</div><div>  B(int);</div><div>  B(const B&) = delete; // deleted, but still trivial</div><div>  B(B&&) = delete; // likewise</div><div>  B *p = this;</div><div>};</div><div>B f(B x) {</div>


<div>  return { 1 };</div><div>}</div><div>void g() {</div><div>  f({2});</div><div>}</div><div><br></div><div>Both of these cases can lead to miscompiles with both g++ 4.7 and Clang trunk due to the ABI's rules. Both problems can also arise for classes with no user-declared special members.</div>

<div><br></div><div>The second issue in particular seems unsolvable without a major ABI break or a core language change (we cannot pass types in registers if they have any non-trivial constructors).</div></div></div></blockquote>
<div><br></div></div></div></div>I think a relatively modest core language change would simply be to say that passing or returning a trivially copyable type is allowed to introduce extra copies.  That would make your examples have unspecified behavior.</div>
</blockquote><div><br></div><div>The rule would need more finessing (as proposed, it would be disastrous for 'struct B', and would still require an ABI break for classes which have trivial copy construction but non-trivial copy assignment), but I think something along these lines could be workable.</div>
<div><br></div><div>Suggestion for core language:</div><div><br></div><div>When an object of class type C is passed to or returned from a function, if C has a trivial, accessible copy or move constructor that is not deleted, and has no non-trivial copy constructors, move constructors, nor destructors, implementations are permitted to perform an additional copy or move of the object using the trivial constructor (even if it would not be selected by overload resolution to perform a copy or move of the object). [Note: This latitude is granted to allow objects of class type to be passed to or returned from functions in registers -- end note]<br>
</div><div><br></div><div>Suggestion for Itanium ABI:</div><div><br></div><div>[parameters and return values are passed by address if] the type has a non-trivial copy constructor, move constructor or destructor, or if neither the copy constructor nor the move constructor is public and non-deleted.</div>
