<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 color=navy face=Arial><span style='font-size:10.0pt;font-family:Arial;
color:navy'><o:p> </o:p></span></font></p>

<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 face="Courier New"><span style='font-size:10.0pt;font-family:"Courier New"'>We
are currently trying to port our application to HP-Itanium. Our application
contains (among other things) an implementation for COM, which is heavily
dependent on the virtual table format. The current virtual table format is very
hard for us to work with. Specifically the issue that is bothering us is 'B-4
Function descriptors in vtable' as detailed in <a
href="http://www.codesourcery.com/cxx-abi/cxx-open.html">http://www.codesourcery.com/cxx-abi/cxx-open.html</a><o:p></o:p></span></font></p>

<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 face="Courier New"><span style='font-size:10.0pt;font-family:"Courier New"'><span
style='mso-spacerun:yes'> </span><o:p></o:p></span></font></p>

<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 face="Courier New"><span style='font-size:10.0pt;font-family:"Courier New"'>The
problem we have is that design of COM relies heavily on the 'fact' that virtual
tables contain pointers to functions. I am attaching a simple C/C++ sample that
shows the kind of usage COM tends to do. This sample works for HP on PA-RISC
(using the aCC compiler). If you want, I can provide a working sample for Linux
on IA32 (using gcc 3.2). <o:p></o:p></span></font></p>

<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 face="Courier New"><span style='font-size:10.0pt;font-family:"Courier New"'>The
kind of applications in which COM makes this assumption are:<o:p></o:p></span></font></p>

<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 face="Courier New"><span style='font-size:10.0pt;font-family:"Courier New"'>1.<span
style='mso-tab-count:1'>    </span>Declaring, as a C literal, a
structure that has the same binary layout as a C++ virtual table.<o:p></o:p></span></font></p>

<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 face="Courier New"><span style='font-size:10.0pt;font-family:"Courier New"'>2.<span
style='mso-tab-count:1'>    </span>Calling a C++ virtual method
from within C code.<o:p></o:p></span></font></p>

<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 face="Courier New"><span style='font-size:10.0pt;font-family:"Courier New"'>Specifically
the second problem is more troubling. It is very common in COM code to see code
like<o:p></o:p></span></font></p>

<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 face="Courier New"><span style='font-size:10.0pt;font-family:"Courier New"'><span
style='mso-spacerun:yes'>  </span>hr =
pObject->lpVtable->method(pObject);<o:p></o:p></span></font></p>

<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 face="Courier New"><span style='font-size:10.0pt;font-family:"Courier New"'><span
style='mso-spacerun:yes'> </span><o:p></o:p></span></font></p>

<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 face="Courier New"><span style='font-size:10.0pt;font-family:"Courier New"'>With
the current vtable layout, every function call like this has to be modified to <o:p></o:p></span></font></p>

<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 face="Courier New"><span style='font-size:10.0pt;font-family:"Courier New"'><span
style='mso-spacerun:yes'>  </span>( (int (*) (IObject)) &
pObject->lpVtable->method ) (pObject);<o:p></o:p></span></font></p>

<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 face="Courier New"><span style='font-size:10.0pt;font-family:"Courier New"'><span
style='mso-spacerun:yes'> </span><o:p></o:p></span></font></p>

<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 face="Courier New"><span style='font-size:10.0pt;font-family:"Courier New"'>This
modification is very hard to do, and is also unique to this specific ABI. The
result is that this specific ABI is designed such that it is very hard to get
COM to work on this ABI.<o:p></o:p></span></font></p>

<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 face="Courier New"><span style='font-size:10.0pt;font-family:"Courier New"'><span
style='mso-spacerun:yes'> </span><o:p></o:p></span></font></p>

<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 face="Courier New"><span style='font-size:10.0pt;font-family:"Courier New"'>I
have seen that this issue has been raised before, without a real solution
offered, at <a href="http://www.codesourcery.com/cxx-abi/cxx-closed.html#B4">http://www.codesourcery.com/cxx-abi/cxx-closed.html#B4</a><o:p></o:p></span></font></p>

<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 face="Courier New"><span style='font-size:10.0pt;font-family:"Courier New"'><span
style='mso-spacerun:yes'> </span><o:p></o:p></span></font></p>

<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><font
size=2 face="Courier New"><span style='font-size:10.0pt;font-family:"Courier New"'>On
possible way to extend the ABI (in order to limit binary compatibility issues)
is to define that the function descriptor can either contain a <function
address/GP> or a <function pointer/-1>. Code that calls a virtual
function will know how to deal with both formats. The compiler can create
either one (maybe dependent on a flag). <o:p></o:p></span></font></p>
