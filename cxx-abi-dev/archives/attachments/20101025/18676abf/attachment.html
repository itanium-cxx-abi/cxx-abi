<p>Hi all, so does anyone know which way this mangling will become? Will it just use the DT/Dt codes or be collapsed. If we have two ways then it clearly will not be binary compatible.<br>
We have also noted the following similar case which seems to be inconsistent.<br>
<tt><font size="4">struct A<br>
{<br>
                 struct B<br>
                 {<br>
                 } b;<br>
}a;<br>
 <br>
template <class X> auto fn1(X x)->decltype(x) {}<br>
template <class X> auto fn2(X x)->decltype(x) {}<br>
template <class X> auto fn3(X x)->decltype(x.b) {}<br>
template <class X> auto fn4(X &x)->decltype(x) {}<br>
 <br>
int main()<br>
{<br>
  fn1(100);                 //_Z3fn1IiET_                    <-- decltype is folded away (should it be preserved?)<br>
  fn2(a);                           //_Z3fn2I1AET_                          <-- decltype is folded away (should it be preserved?)<br>
  fn3(a);                           //_Z3fn3I1AEDtdtfp_1bET_<br>
  fn4(a);              //_Z3fn4I1AEDtfp_ <br>
}</font></tt><br>
<br>
<font size="4">In these special cases, what is the rationale behind folding away decltype mangling for dependent types?  </font><br>
<font size="4">As a follow-up, are there special cases where decltype mangling is preserved for: decltype expressions with non-dependent types, value-dependent decltype expressions? </font><br>
